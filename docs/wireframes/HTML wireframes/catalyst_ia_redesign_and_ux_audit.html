Catalyst UX audit and revised information architecture with document workspace and RAG search placement

Honest assessment of what works and what doesn't in the current design, given that manual findings and notes will be the primary workflow for a while.

friction "Add finding" is buried
The button lives at the top of the center panel in the main case view — but only when you're on the Overview tab. If you're reading a document on the Documents tab and spot something suspicious, you have to navigate back to Overview to add a finding. That's one extra tab click every time you want to record something, which adds up fast.
Fix: persistent "Add finding" shortcut visible from any tab
friction Notes and findings are two separate things right now
The current design has InvestigatorNote (informal, no signal rule) and Finding (formal, citable). But in practice, the investigator workflow is: notice something → jot a note → promote it to a finding when there's enough evidence. There's no clear path from note to finding in the current design. They live in different places and don't visually connect.
Fix: notes as "draft findings" — same panel, lighter styling, one "Promote to finding" button
friction 6 tabs is one too many
Overview, Documents, Research, Financials, Timeline, Referrals = 6 tabs. The cognitive load isn't the count, it's that "Overview" and "Documents" feel like they should be unified — right now Overview has the investigation workspace but no document viewer, and Documents has a file list but no investigation context. They need each other.
Fix: merge Overview + Documents into one "Investigate" workspace (detailed below)
friction No cross-document search anywhere in the flow
You're reading a 990 and you see "Baumer Construction." To find every other document that mentions Baumer, you have to navigate to the global search, type the name, get results, click back into a document. There's no way to search while staying in context of what you're reading. This is a real investigation workflow — you follow threads — and the current design breaks it.
Fix: RAG search sidebar on any open document (detailed below)
works Three-panel layout is the right foundation
The left rail / center / right panel structure is correct. The investigation context (who, what, connects to what) in the right panel is genuinely useful. Keep this — but it needs to be accessible from more places, especially from inside a document.
works Research, Financials, Timeline as supporting tabs
These three tabs are correctly scoped — they're lookup/analysis surfaces, not the primary investigation workspace. The navigation to them is fine. The issue is getting back from them into the investigation flow (a finding you discover in Financials should have a one-click path back to the center panel).

Proposed change: collapse 6 tabs to 5 by merging Overview + Documents into one "Investigate" tab. Everything else stays the same.

"Investigate" combines the three-panel findings workspace with a full document viewer. Left rail = document list (+ upload). Center = document viewer OR findings list (toggle). Right = context panel (findings from this doc, or finding detail). RAG search lives here as a collapsible sidebar.

Investigate tab — what lives here

  • All case documents (upload, view, OCR)
  • All findings + notes (list, add, promote)
  • Document viewer with inline finding highlights
  • RAG cross-document search sidebar
  • Entity-relationship graph (collapsed by default)
  • "Add finding" always accessible

Supporting tabs — unchanged scope

  • Research — external data sources (IRS, AOS, SOS, Recorder)
  • Financials — 990 year-over-year table + anomaly flags
  • Timeline — chronological event view
  • Referrals — export checklist + PDF generation
The key change: "Add finding" and the document list live in the same tab. You read a document, see something, click "Add finding" — without ever leaving the page. The note/finding distinction is handled by evidence weight, not by two separate UI objects.

Left rail shows all documents with finding counts — documents with no findings yet get a warning badge so nothing gets overlooked. Findings are also listed at the bottom of the left rail so the full picture is always one glance away. The "Add finding" button lives in both the top toolbar and the right panel — always within reach.

RAG search (searching other documents for related content while reading one document) lives as a collapsible sidebar within the Investigate tab. It does not require a new page.

How RAG search works here

  • Searches only documents uploaded to this case — not all cases
  • Highlight text → right-click → "Search other docs for this"
  • Or use the search bar at the top of the Investigate tab
  • Results show an excerpt with the matching passage + which document + which page
  • Each result has "Add finding" so you can act immediately without opening the doc
  • Results do NOT replace the document you're reading — sidebar only

What this requires on the backend

  • OCR must have run on all documents (already in the pipeline)
  • Each document's text needs to be chunked and indexed per case
  • Search can be keyword (fast, works now) or semantic/embedding (slower, more powerful)
  • Start with keyword search — grep the extracted text. Ship that. Upgrade to embedding search later as a YELLOW decision.
  • Backend endpoint: GET /api/cases/<id>/doc-search/?q=Baumer+Construction
