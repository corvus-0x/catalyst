These are the edge cases and validation states both modals need to handle correctly. Getting these right is what separates a form that works from one that loses data or confuses the investigator mid-workflow.

Validation states
!
Empty required field on save — border turns red, label shows error text inline. No toast. Scroll to first error.
↓
Weight="Traced" with no doc — auto-demotes to "Documented" silently when doc is removed. Shows "Evidence weight updated" toast.
→
Rule auto-fill — selecting a signal rule pre-fills severity. Investigator can still override — the rule is a suggestion, not a lock.
✓
Duplicate detection — if title text closely matches an existing finding, show "Possible duplicate: [Finding name]" link below title field.
Abandon / discard logic
1
Form is empty — pressing ✕ or Escape closes immediately, no confirmation
2
Form has content but not saved — pressing ✕ shows: "Discard changes? / Cancel · Discard"
3
"Save as draft" path — always available on Add Finding drawer even with incomplete data. Saves status=NEW, weight=SPECULATIVE.
4
Back navigation (Add Event) — going back preserves all entered data in that step; never clears on back
The "create finding from event" chain
When investigator picks "Create new finding from this event" in step 3 of Add Event:
Add event modal Saves event to timeline
↓ modal closes
Add finding drawer Opens with event data pre-filled
↓ investigator fills description + weight only
Finding saved Auto-linked back to timeline event
Keyboard / accessibility
Escape — closes drawer/modal (triggers discard check if dirty)
Tab order — title → rule/type → severity → weight → description → documents → entities → note → save
Cmd/Ctrl + Enter — triggers "Save finding" / "Next" from any field
Drawer backdrop — clicking the dimmed main case view behind the drawer does NOT close it (investigator might be reading context)
Modal backdrop — clicking outside the Add Event modal DOES trigger the discard check (it's a true modal, not a slide-in)
