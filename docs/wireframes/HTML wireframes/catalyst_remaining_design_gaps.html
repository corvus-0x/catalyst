Blockers — design these before writing any code
1
The web view — visualizing knots connected by threads
Tyler just confirmed this is a web, not a single-point investigation. Karen Homan, Hanes Law Group, Do Good Inc, Baumer Construction — all knots. The thread between Hanes and Homan (LLC assembly) is an edge between two knots. We need a view that shows this web. Without it, the investigator can't see the shape of the case.
Why blocker: the Investigate tab's knot view only works if you have one knot. As soon as there are multiple, you need the web view to navigate between them and see cross-knot threads.
no design yet
2
The Referrals tab — the actual product output
This is the whole point of Catalyst. The referral package PDF is the deliverable. We haven't designed: what the pre-export checklist looks like, how confirmed threads from multiple knots get organized into the package, the review step before Tyler clicks "Generate PDF." The backend exporter already exists — it just needs a UI that makes it trustworthy to use.
Why blocker: this is what a recruiter or interviewer will ask about. "Walk me through generating a referral package" needs a clear, designed answer.
no design yet
3
Evidence confirmation — reviewing what Haiku extracted
Haiku runs on every uploaded document and surfaces candidate facts. But we haven't designed where those candidates live or how Tyler reviews and confirms them. Right now they'd just silently appear on a thread — but some will be wrong (wrong name match, OCR noise, hallucinated EIN). The investigator needs a simple "accept / reject" flow. Rejected evidence should be kept but flagged as dismissed — not deleted, because chain of custody.
Why blocker: without this, Haiku output is trusted by default, which is dangerous for a referral package that will be read by an AG investigator.
no design yet
Important — design in this session or the next
4
Thread splitting — one question becomes two
We agreed this is needed. We haven't designed the actual interaction. When Tyler is pulling the hidden compensation thread and realizes it has split into two separate questions (off-book pay vs. contractor fraud), how does he divide the evidence? Does he drag evidence items to the new thread? Does he pick which documents belong to each child thread? This needs a concrete flow.
This happens in real investigations. The Do Good case is full of it — the Hanes LLC thread split into "who assembled the entities" and "what's the relationship between the two Do Good LLCs."
needs design
5
Quick capture — noting something before it has a thread
Right now you can only add evidence to an existing thread. But sometimes you're reading a document and something catches your eye that doesn't fit any current thread — and you don't want to stop and create a full thread just to note it. You need a fast "flag this and come back to it" action. Something like a floating sticky note that lives in an "unassigned evidence" pile until you route it to a thread.
This is the digital equivalent of a sticky note on a physical document. Investigators work this way. Without it, things get lost.
needs design
6
Case readiness check — is this ready to refer?
Before generating the referral package, there should be a view that shows: how many threads are confirmed vs. still pulling, which confirmed findings have citations vs. which don't, whether the key entities have been identified, whether the 990 data has been fetched. This is the "pre-flight checklist" — it tells Tyler what's missing before he commits to generating the PDF.
The quality bar is "heavy confidence that it was going to go somewhere." This view is how you verify that bar has been met.
needs design
