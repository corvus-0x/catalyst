## 0. Local loop & baseline

### 0.1 Local DB reality

**What actually serves the local database:** a **native Windows PostgreSQL 18**
install — Windows service `postgresql-x64-18` (State: Running, StartMode: Auto,
data dir `C:\Program Files\PostgreSQL\18\data`, binary
`C:\Program Files\PostgreSQL\18\bin\pg_ctl.exe`). The postmaster (PID 11252) is the
sole listener on **port 5433** (`0.0.0.0:5433`). Nothing listens on 5432 or 5434.

**Docker is NOT running** — `docker ps` fails with
`failed to connect to the docker API ... dockerDesktopLinuxEngine`. So the
`catalyst_db` Docker container that STATUS.md and the status memory describe
("PostgreSQL in Docker container `catalyst_db` (must be running first)") is **not**
the database in use. The docs drifted from reality: the DB moved to a native PG18
install and the docs were never updated. → **doc-drift punch-list item.**

**The `.env` is configured for the dead Docker setup, not the live native PG18:**

| `.env` key | Value | Reality | Verdict |
|------------|-------|---------|---------|
| `DB_HOST` | `127.0.0.1` | correct | ✅ |
| `DB_PORT` | **`5434`** | live Postgres is on **5433**; nothing on 5434 | ❌ wrong port |
| `POSTGRES_USER` | `postgres` | user exists on PG18 | ✅ (user) |
| `DB_PASSWORD` | (8 chars) | **fails auth** for `postgres` on the native PG18 | ❌ wrong password |
| `POSTGRES_DB` | `catalyst_db` | not yet confirmed to exist on PG18 (couldn't auth) | ⚠️ unverified |

**Consequence:** with `.env` as-is, `manage.py runserver` / `test` / `qcluster` all
aim at a dead port (5434) and, even when pointed at 5433, fail password auth. This is
the complete root cause of "can't run locally → push to Railway to find out." It is a
**config-drift** issue, not a code bug. → **P0 punch-list item** (see §3.1).

**Verified live (2026-06-03):**

- `Get-NetTCPConnection -LocalPort 5433` → owned by PID 11252 (postgres postmaster).
- `Get-CimInstance Win32_Service ...postgres...` → `postgresql-x64-18`, Running, Auto.
- `manage.py test ...test_searchjob_model` with `DB_PORT=5433` →
  `OperationalError: FATAL: password authentication failed for user "postgres"`.

**Resolution (2026-06-03):** Tyler chose `trust` auth for the local dev DB. Applied:
backed up `pg_hba.conf` → `pg_hba.conf.bak-20260603`, replaced `scram-sha-256` → `trust`

applied migrations 0001→0034, created and destroyed `test_catalyst_db`, 4/4 OK. The local DB
path is unblocked. (Security: `trust` is local-only and reversible via the `.bak`. Never use
on a network-exposed/prod DB.) The stale Docker-era `.env`/STATUS.md DB description remains a
doc-drift punch-list item.

**Docker reconciliation (2026-06-03, Docker Desktop started mid-audit):** Three DB

**Two missing local-setup steps surfaced (the real "why I couldn't run locally"):**

2. **Cache table missing.** `qcluster` spammed
   `relation "catalyst_cache_table" does not exist` — Django uses a **database cache backend**

   worker self-recovered (errors stopped within seconds, no restart needed).

Together with §0.1 (trust auth + `DB_PORT` fix), the **complete first-run setup** is: fix DB
auth/port → `migrate` → `createcachetable` → (optional `seed_demo`) → start the three tiers.
